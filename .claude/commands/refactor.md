TDD Refactor Phase: 모든 테스트가 통과하는 상태에서 코드 품질을 개선한다.

리팩토링 대상/방향: $ARGUMENTS

## 절차

1. 전체 테스트가 통과하는 상태인지 확인한다: `./gradlew test`
   - 실패하는 테스트가 있으면 리팩토링을 중단하고 보고한다
2. `$ARGUMENTS`가 있으면 해당 대상/방향을 우선 처리한다
3. `$ARGUMENTS`가 없으면 최근 변경된 코드를 분석해 개선점을 제안한다
4. 리팩토링을 수행한다 (한 번에 하나의 리팩토링):
   - 중복 제거
   - 네이밍 개선 (이름과 구조로 의도를 명확히 표현)
   - 메서드 추출 (단일 책임 원칙)
   - 불필요한 private 함수 지양, 객체지향적 코드 작성
   - 의존성을 명시적으로 드러내기
   - unused import 제거
5. 린트를 수정한다: `./gradlew ktlintFormat`
6. 린트 체크를 확인한다: `./gradlew ktlintCheck`
7. 전체 테스트를 재확인한다: `./gradlew test`
8. 결과를 보고한다:
   - 변경 내용 요약
   - 변경된 파일 목록
   - 린트 결과
   - 테스트 결과

## 규칙

- 테스트가 모두 통과하는 상태에서만 리팩토링
- 리팩토링은 **구조적 변경만** (동작 변경 금지)
- 한 번에 하나의 리팩토링 수행, 매 단계마다 테스트 실행
- 리팩토링 후 테스트가 실패하면 변경을 되돌리고 보고한다
- 가능한 가장 단순한 해결책을 사용한다
