Tidy First 규칙에 따라 커밋을 생성한다.

## 절차

1. `./gradlew ktlintCheck test`를 실행해 린트 + 테스트 통과를 확인한다
   - 통과하지 않으면 커밋을 중단하고 실패 내용을 보고한다
2. `git status`와 `git diff --staged`, `git diff`로 변경 내용을 분석한다
3. `git log --oneline -10`으로 최근 커밋 메시지 스타일을 확인한다
4. 변경 유형을 판단한다:
   - **구조적 변경**: 동작을 바꾸지 않는 코드 재배치 (이름 변경, 메서드 추출, 코드 이동, unused import 제거)
   - **행위적 변경**: 실제 기능 추가/수정 (새 기능, 버그 수정, 테스트 추가)
5. 구조적 변경과 행위적 변경이 섞여 있으면:
   - 개발자에게 분리를 제안한다
   - 분리 방법을 구체적으로 안내한다 (어떤 파일이 구조적이고 어떤 파일이 행위적인지)
6. 커밋 메시지를 한국어로 작성한다
7. 개발자에게 커밋 메시지를 확인받은 후 커밋한다

## 커밋 메시지 형식

```
유형: 변경 내용 요약

상세 설명 (필요한 경우)
```

### 유형 접두사

- `feat:` — 새 기능 추가
- `refactor:` — 동작 변경 없는 코드 재배치 (구조적 변경)
- `fix:` — 버그 수정
- `test:` — 테스트 추가/수정
- `chore:` — 설정 파일, 빌드 스크립트 변경
- `docs:` — 문서 변경

### 예시

```
feat: 회원가입 시 로그인 ID 중복 검증 추가

- UserService에 중복 체크 로직 구현
- UserRepository에 existsByLoginId 메서드 추가
```

```
refactor: UserService 메서드 추출로 가독성 개선

- validateUser 메서드를 validateLoginId, validatePassword로 분리
```

```
test: 회원가입 실패 케이스 단위 테스트 추가
```

## 규칙

- 모든 테스트가 통과하고, 린터 경고가 없을 때만 커밋
- 하나의 논리적 작업 단위로 커밋
- 크고 드문 커밋보다 작고 빈번한 커밋을 지향
- 구조적 변경과 행위적 변경을 절대 같은 커밋에 섞지 않는다
